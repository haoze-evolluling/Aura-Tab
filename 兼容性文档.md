## 追求最高兼容性的项目设计与执行方案

### I. 基础技术与兼容性原则

为保证在所有主流浏览器（Chrome、Firefox、Edge、Safari，包括部分旧版本）上都能良好运行，我们将遵循以下原则：

1.  **HTML5 语义化结构：** 仅使用 `<header>`、`<main>`、`<footer>` 等基础且稳定的标签构建骨架。
2.  **原生 CSS3 基线：** 避免使用 CSS 变量（`var()`），放弃最新的 Grid 布局，转而使用**Flexbox**作为主要的排版工具，并提供基础的**浮动（Float）**回退方案（但Flexbox本身已兼容良好）。
3.  **ES5/ES6 兼容代码：** 核心 JavaScript 代码应以 **ES5** 语法为基础，并谨慎使用 ES6 新特性。如果使用了箭头函数或 `const/let`，我们会依赖现代浏览器环境或考虑使用 Babel 进行转译（尽管这会增加构建复杂度，但可确保兼容性）。

### II. 分布式代码的兼容性拆分与引入

为了实现分布式（模块化），我们将使用最传统的、兼容性最好的方式来引入文件，而不是依赖新的`type="module"`语法。

#### 1. CSS 模块化与兼容性引入

所有 CSS 文件将保持独立，并通过标准的 `<link>` 标签在 HTML `<head>` 中引入：

* **`styles/reset.css`：** 采用经典的 Eric Meyer Reset 或 Normalize.css，解决浏览器默认样式差异。
* **`styles/base.css`：** 定义全局字体、颜色和背景。字体仅使用 **系统默认字体栈**（如 `Arial, sans-serif`），以避免加载外部字体带来的兼容性和性能问题。
* **`styles/layout.css`：** 使用 **Flexbox** 实现核心模块的居中对齐。对于旧版浏览器，使用 `text-align: center` 和固定的 `width` 作为降级方案。
* **`styles/components.css`：** 集中放置搜索框、常用网站等组件的样式，避免复杂的伪类和过渡效果。

#### 2. JavaScript 模块化与兼容性引入

JS 模块通过标准的 `<script>` 标签按顺序引入，确保依赖关系正确，并全部放在 `<body>` 结束标签前，以实现**异步加载**，优化感知速度。

* **`scripts/utils.js`：** 放置所有工具函数，如 DOM 选择器封装、`localStorage` 读写封装。代码必须遵循 ES5 规范。
* **`scripts/time-module.js`：** 负责时钟逻辑。使用传统的 `function` 声明，并用 `Date()` 对象实现实时时间更新。
* **`scripts/search-module.js`：** 负责处理搜索表单提交。使用 `document.getElementById` 和 `addEventListener`，避免复杂的框架或库。
* **`scripts/main.js`：** 作为项目入口，负责依次调用和初始化所有模块的函数。

### III. HTML 结构与降级处理

* **结构：** HTML 将保持极简，只作为内容的容器。所有功能元素（如搜索框、时间显示区、链接区）都拥有唯一的 **ID**，方便 JavaScript 准确操作。
* **降级：** 常用网站链接（Quick Links）即使 JavaScript 失败，也应在 HTML 中包含**至少一个**默认的静态链接列表，确保在最差兼容环境下，用户仍能点击核心网站，不至于白屏或完全不可用。

### IV. 执行关键点：避免使用

为了最大化兼容性，在执行过程中必须严格避免以下特性：

1.  **CSS 变量 (`--var`)** 和 **`calc()`** 复杂运算。
2.  **Grid 布局**和复杂的 3D **`transform`** 动画。
3.  **JS `fetch` API**（尽管兼容性不错，但为保险起见，若有外部数据请求，应考虑使用 **`XMLHttpRequest`** 作为兼容性回退）。
4.  所有依赖 **Promises** 或 **Async/Await** 的复杂异步逻辑，将其转换为传统的回调函数模式。

通过这种“退回到稳定基线”的设计方法，我们能确保新标签页在最广泛的浏览器环境中都能快速、可靠地工作。
